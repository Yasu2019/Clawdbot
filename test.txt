# 前提として分かっている不具合
以下は既に見つかっているため、必ず確認・修正対象に含めること。

## 1. CSVインポート仕様の不整合
- 現行の import_test は headers: true 前提で CSV.foreach している
- しかし、生成済みCSVはヘッダー無しのデータ行始まりが多い
- このため、そのままでは正しく import できない可能性が高い

## 2. CSV列構造の不整合
DB/モデル側の想定項目は実質的に以下であることを前提に確認すること。
- kajyou
- mondai_no
- rev
- mondai
- mondai_a
- mondai_b
- mondai_c
- seikai
- kaisetsu

必要に応じて id を扱う場合は明確化すること。

一方、CSV側には以下の揺れがある。
- 9列以外のCSVが存在する
- 10列、11列、6列、17列などが混在している
- 「適合例」「不適合例」「解説」が別列化されているものがある
- 選択肢が問題文に埋め込まれており、mondai_a/b/c に分かれていないものがある
- 空行がある
- 20問固定でないCSVがある

## 3. 正解率ロジックの不整合
- seikai は a/b/c の文字列なのに、正解件数集計で seikai: true のような比較があり不正確
- seikairitsu が保存時は 0〜1、表示時は 0〜100 になっており単位が不一致の可能性がある

## 4. 文字数問題
- migration上、mondai や kaisetsu が string の可能性がある
- 解説文は 255文字を超えるケースがあり、DB切り捨てや保存失敗の危険がある

## 5. 取込導線の確認
- ZIP一括取込ではなく、CSV単体取込UIの可能性がある
- 実際の管理画面・controller・view・model の流れを確認し、仕様を明文化すること

## Phase 1: 現状調査

## Phase 2: 修正実装
以下を実装してください。

### A. CSV取込の堅牢化
- ヘッダー有り/無しの両方に対応すること
- 列構造の揺れを吸収する importer / normalizer を作ること
- 少なくとも以下の標準形に正規化すること

  kajyou, mondai_no, rev, mondai, mondai_a, mondai_b, mondai_c, seikai, kaisetsu

- 余分な列がある場合は、ルールを定義して kaisetsu に統合するなど、意味を失わず正規化すること
- 選択肢が問題文に埋め込まれているCSVは、可能なら安全に分解すること
- 分解不能なCSVはスキップではなく、エラー内容を明示してレポートすること
- 空行は無視すること
- 不正行は何行目の何が不正かを分かるようにすること

### B. 正解率ロジック修正
- seikai は a/b/c の文字列として扱うこと
- 正解件数の集計ロジックを修正すること
- seikairitsu の保存単位を統一すること
- 推奨は「DB保存は 0〜100 の百分率」または「0.0〜1.0 の比率」のどちらかに統一し、
  どちらにしたかを明示すること
- 画面表示・Excel出力・集計処理で単位不一致が起きないようにすること

### C. DB型の見直し
- mondai, kaisetsu は text 型への変更を検討し、必要なら migration を追加すること
- 既存データ移行時に壊れないよう配慮すること

### D. 出題・採点・結果表示の整合確認
- 問題表示画面
- 回答送信
- 正誤判定
- 正解率集計
- 結果表示
- Excel出力やCSV出力があればそれも
これら一連の流れが破綻しないことを確認すること

## Phase 3: テスト追加
最低でも以下のテストを追加してください。
1. ヘッダー有りCSVの取込
2. ヘッダー無しCSVの取込
3. 9列CSVの取込
4. 余分列付きCSVの取込
5. 不正列CSVのエラーハンドリング
6. a/b/c 正答判定
7. 正解率算出
8. 長文解説の保存

RSpec / Minitest は現行構成に合わせてください。
