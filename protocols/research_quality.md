# Research Quality

## Purpose
- 品証、不具合調査、規格確認、顧客要求確認を、推定と事実を混同せず整理する。

## Input
- 対象メール、図面、規格書、監査メモ、現品写真、測定結果、過去不具合。
- メール起点の場合は先に `email_search_query.py` または `tasks-context` を引く。

## Procedure
1. 問題定義を 1 文で確定する。
2. 対象品番、ロット、工程、仕入先、発生日、検出日を整理する。
3. 事実、推定、未確認事項を分離する。
4. 規格、図面、顧客要求、社内基準のどれに対する差異かを切り分ける。
5. リスクを品質、納期、顧客影響、横展開要否で評価する。
6. 不足証拠と追加確認事項を列挙する。
7. 結論を、現時点で言えることだけでまとめる。

## Required Fields
- 問題定義
- 対象品番 / ロット / 工程 / 仕入先
- 発生点 / 検出点
- 事実
- 推定
- 未確認事項
- 必要証拠
- 規格 / 図面 / 顧客要求 / 社内基準の切り分け
- リスク評価
- 追加確認事項
- 結論

## Rules
- 事実と推定を同じ箇条書きに混ぜない。
- 根拠がメールの場合は message source を残す。
- 証拠が不足しているときは、断定しない。
- 顧客要求と社内運用ルールを混同しない。
