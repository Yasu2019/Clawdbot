# Self-Healing Boundary

Last updated: 2026-03-29 JST

## 結論

この環境では、`承認が必要だから直せなかった` ケースは多くありません。
実際には、次の3種類があります。

1. 自動で検知して、そのまま修復できるもの
2. 自動で検知できるが、修復方法に分岐があり判断が必要なもの
3. 外部要因や業務判断が絡み、人が最終判断した方が安全なもの

今回の実態は、`1` は一部動いていたが、`2` と `3` の扱いが曖昧だった、という状態でした。

## 自動修復できる範囲

以下は、原則として自動で回復対象にできます。

- status JSON が stale になった定期同期の再実行
- SQLite の WAL 非対応や共有ドライブ I/O 問題の回避
- `localhost` / `host.docker.internal` / `127.0.0.1` の接続先フォールバック
- timeout が短すぎる処理の実行時間見直し
- 重複した外付けハーネスの古い Python プロセス停止
- `learning_engine` のような補助サービスの再ビルド・再起動
- UI の軽微な表示不整合修正

## 自動検知できるが判断が必要な範囲

以下は、検知までは自動でできても、修復方針に複数案があります。

- CORS を最小許可にするか広く許可するか
- timeout をどこまで伸ばすか
- nightly を即再実行するか、次回 scheduled run まで待つか
- stale status を上書きするか、履歴として残すか
- Mini PC 最適化のために、どの常駐サービスを止めるか

これらは技術的には私がかなり対応できますが、運用方針次第で最適解が変わります。

## 人の判断が残る範囲

以下は、承認というより業務判断や外部条件の問題です。

- 商用 SDK 導入
- 外部アカウント、認証、課金、API キーの変更
- どのデータを cross-org で使ってよいか
- 既存運用を止める可能性がある大規模な compose 整理
- どの UI / アプリを正式運用対象にするか

## 今回わかったこと

今回の不具合群から分かったのは次の通りです。

- 監視は存在していた
- stale 判定や status 出力も存在していた
- ただし、修復ロジックが「再実行」中心で、構成変更を伴う不具合までは届いていなかった
- そのため、`自律運転` というより `半自律 + 人の介入前提` だった

## 今後の方針

この環境では、次の考え方が現実的です。

- 原則: 自動検知
- 原則: 低リスク修復は自動実行
- 条件付き: 影響範囲が広い修復は status と理由を残してから実行
- 人判断: 費用、権限、法務、業務フロー変更が絡むもの

## 今回すでに自動化へ近づけた項目

- Learning Console の API 接続復旧
- learning_engine CORS 調整
- Email Learning sync の重複停止と軽量化
- scheduled report sync の temp DB 化
- CAE sync の base URL フォールバック
- nightly timeout の延長

## ひとことで言うと

`人手が必要` とは、必ずしも「承認ボタンが必要」という意味ではありません。
多くは、`どの修復方針を採るべきかの判断が必要` という意味です。

あなたが今回のように包括的に進行許可を出してくれている範囲では、かなりの部分を私がそのまま修正できます。
